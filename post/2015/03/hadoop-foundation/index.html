1. What Is Apache Hadoop?

The Apache™ Hadoop® project develops open-source software for reliable, scalable, distributed computing.

The Apache Hadoop software library is a framework that allows for the distributed processing of large data sets across clusters of computers using simple programming models. It is designed to scale up from single servers to thousands of machines,s each offering local computation and storage. Rather than rely on hardware to deliver high-availability, the library itself is designed to detect and handle failures at the application layer, so delivering a highly-available service on top of a cluster of computers, each of which may be prone to failures.

The project includes these modules:

  • Hadoop Common: The common utilities that support the other Hadoop modules.
  • Hadoop Distributed File System (HDFS™): A distributed file system that provides high-throughput access to application data.
  • Hadoop YARN: A framework for job scheduling and cluster resource management.
  • Hadoop MapReduce: A YARN-based system for parallel processing of large data sets.

4. QJM HA的原理

使用QJM（Quorum Journal Manager）实现HDFS的HA，同时只有一个NameNode节点是active，另一个节点是standby。为了保持standby节点和active节点的状态同步，两个NameNode节点通过一组叫做JournalNodes(JNs)的节点通信，active节点执行任何namespace的修改都会记录一条日志到JournalNodes 过半数节点的edits目录（参见hdfs-site.xml的dfs.journalnode.edits.dir配置）。standby能够读取这些变化，定期查看edit log并将变化部分应用到自己的namespace。Standby节点持有的集群block信息必须是最新的才能保证成功的failover，所以为了完成failover，DataNode的配置要包括active和standby的位置，DataNode给active和standby节点都发送block location信息，并执行心跳检测。

任一时刻只有一个Namenode节点是active状态是至关重要的，否则可能产生两份不同的namespace状态信息，加剧数据丢失和错误发生的风险，为了避免这种可能和防止脑裂，同一时刻JournalNodes只允许一个NameNode节点写入数据。发生failover期间，变成active状态的NameNode节点只需简单地接管写入JournalNodes的角色，这样能有效的防止其它节点继续进行active状态转换，并允许新的active节点安全地执行failover。

5. QJM自动failover原理

Automatic failover adds two new components to an HDFS deployment: a ZooKeeper quorum, and the ZKFailoverController process (abbreviated as ZKFC).

Apache ZooKeeper is a highly available service for maintaining small amounts of coordination data, notifying clients of changes in that data, and monitoring clients for failures. The implementation of automatic HDFS failover relies on ZooKeeper for the following things:

  • Failure detection - each of the NameNode machines in the cluster maintains a persistent session in ZooKeeper. If the machine crashes, the ZooKeeper session will expire, notifying the other NameNode that a failover should be triggered.
  • Active NameNode election - ZooKeeper provides a simple mechanism to exclusively elect a node as active. If the current active NameNode crashes, another node may take a special exclusive lock in ZooKeeper indicating that it should become the next active.

The ZKFailoverController (ZKFC) is a new component which is a ZooKeeper client which also monitors and manages the state of the NameNode. Each of the machines which runs a NameNode also runs a ZKFC, and that ZKFC is responsible for:

  • Health monitoring - the ZKFC pings its local NameNode on a periodic basis with a health-check command. So long as the NameNode responds in a timely fashion with a healthy status, the ZKFC considers the node healthy. If the node has crashed, frozen, or otherwise entered an unhealthy state, the health monitor will mark it as unhealthy.
  • ZooKeeper session management - when the local NameNode is healthy, the ZKFC holds a session open in ZooKeeper. If the local NameNode is active, it also holds a special “lock” znode. This lock uses ZooKeeper’s support for “ephemeral” nodes; if the session expires, the lock node will be automatically deleted.
  • ZooKeeper-based election - if the local NameNode is healthy, and the ZKFC sees that no other node currently holds the lock znode, it will itself try to acquire the lock. If it succeeds, then it has “won the election”, and is responsible for running a failover to make its local NameNode active. The failover process is similar to the manual failover described above: first, the previous active is fenced if necessary, and then the local NameNode transitions to active state.

6. Hbase为什么需要hdfs-site.xml

Hbase是HDFS的一个client，Hbase的数据存储在HDFS上。有些属性既可以在Client端（Hbase）设置，也可以在Server端（HDFS）设置，例如dfs.replication。那么当Client和Server端的设置不一样时，应该使用哪个值？为了解决这个问题，Hbase提供3种方法保证优先使用Hbase中的设置（参照官方Getting Started文档的HDFS Client Configuration部分）。使用如下其中一种方法就可以，其目的都是让Hbase可以读取重复配置，优先使用Hbase中的配置。

  • Add a pointer to your HADOOP_CONF_DIR to the HBASE_CLASSPATH environment variable in hbase-env.sh.
  • Add a copy of hdfs-site.xml (or hadoop-site.xml) or, better, symlinks, under ${HBASE_HOME}/conf.
  • if only a small set of HDFS client configurations, add them to hbase-site.xml.

例如，Hbase设置dfs.replication的值是5，说明用户希望数据存储5份，但是HDFS中这个值默认是3，如果你不按如上方法配置，那么实际上数据存储了3份而不是5份，和用户的期望不一样。

另外，通过环境搭建发现这已经成为必须设置了，如果不按如上方法处理，start-hbase.sh执行后，HRegionServer会报错无法启动。

8. HDFS Federation架构

Federation解决了NameNode不能扩展的问题，Federation并不代表HA，仅仅是表示一个HDFS上可以有多个NameNode对外提供服务，每个NameNode既可以是单点的，也可以利用QJM等方案实现Active/Standby的HA。结合HA方案和Federation可以避免单点故障并实现NameNode横向扩展，实现过程参照Federation + HA + Hbase搭建。

Block Pool

A Block Pool is a set of blocks that belong to a single namespace. Datanodes store blocks for all the block pools in the cluster. It is managed independently of other block pools. This allows a namespace to generate Block IDs for new blocks without the need for coordination with the other namespaces. The failure of a Namenode does not prevent the datanode from serving other Namenodes in the cluster.

A Namespace and its block pool together are called Namespace Volume. It is a self-contained unit of management. When a Namenode/namespace is deleted, the corresponding block pool at the datanodes is deleted. Each namespace volume is upgraded as a unit, during cluster upgrade.

ClusterID

A new identifier ClusterID is added to identify all the nodes in the cluster. When a Namenode is formatted, this identifier is provided or auto generated. This ID should be used for formatting the other Namenodes into the cluster.

Key Benefits

  • Namespace Scalability - HDFS cluster storage scales horizontally but the namespace does not. Large deployments or deployments using lot of small files benefit from scaling the namespace by adding more Namenodes to the cluster
  • Performance - File system operation throughput is limited by a single Namenode in the prior architecture. Adding more Namenodes to the cluster scales the file system read/write operations throughput.
  • Isolation - A single Namenode offers no isolation in multi user environment. An experimental application can overload the Namenode and slow down production critical applications. With multiple Namenodes, different categories of applications and users can be isolated to different namespaces.

9. 理解viewFS

HDFS中的文件/目录操作类似于linux，都是从根目录开始，在Federation出现之前，每个HDFS只有一个NameNode对外提供服务，用户只能一个根目录执行文件/目录操作，因此不会出现目录冲突的情况。在出现Federation后会有多个NameNode对外提供服务，这些NameNode是平等关系，每个NameNode都可以有一个根目录，在这种情况使用传统根目录的方法，HDFS就无法判断一个指定的目录到底是哪个NameNode的，除非使用指定namenode的全路径可以避免这种迷惑，但是操作不友好，意味着每次访问HDFS都得用hdfs:namenode/xxx这种模式，为了继续使用根目录这种操作方式，就需要一个全局统一的namespace管理机制，viewFS就是实现这个目的的。

viewFS相当于linux下的mount table，比如你可以将光驱mount到/media/cdrom，这个目录叫做mount点，我们只需要访问mount点就可以到达真实的光驱。viewFS也起这样的作用，可以实现将一个虚拟目录映射到真实的物理路径，比如/data代表hdfs://cluster01/data，/home代表hdfs://cluster02/home，这样就把多个NameNode区分开来了。
