解决方法

1
2
3
4
5

--在生产库执行下面两个命令就可以了
1. 命令1的目的是使primary解析通向DG库的地址
    alter system set log_archive_dest_2 = 'service="(DESCRIPTION=(ADDRESS=(PROTOCOL=TCP)(HOST=192.168.250.31)(PORT = 1521))(CONNECT_DATA=(SERVER=DEDICATED)(SERVICE_NAME=ctsdb)))" LGWR ASYNC NOAFFIRM  REOPEN=60  db_unique_name=ctsdbdg' ;
2. 命令2的目的是恢复原来的参数设置
    alter system set log_archive_dest_2 = 'service=ctsdbdg LGWR ASYNC NOAFFIRM  REOPEN=60 max_failure=3 net_timeout=30 valid_for=(online_logfiles,primary_role) db_unique_name=ctsdbdg' ;

分析总结

  1. 观察DG库alert日志，发现从2月1号22：57分开始，DG库收不到primary库的任何信息
  2. 观察primary库alert日志，发现LNS进程未发送任何信息给DG库
  3. 通过上面信息，可以判断，肯定是primary和DG的通信有异常
  4. 常见的通信异常原因有防火墙不通、TNS配置错误等，但是验证这些都是正常的
  5. 排除常见原因，目前的可能就是primary根本没有解析到连接DG库的连接串
  6. 因此我通过直接给出IP和端口的方法，让primary硬解析地址，使primary和DG通信
  7. 重新设置dest2参数，切换日志，可以观察到DG库已经收到primary的信息，开始在standby log上同步日志并apply日志
  8. 同样也可以观察到primary库已经启动LNS进程，传输log到DG库
  9. 至此，这个问题就处理完了
