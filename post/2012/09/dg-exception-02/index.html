1. 确认DG目前为mount standby状态
2. SQL> select name from v$datafile;
3. 检查如上查询的文件名，发现和实际的不一致
4. 生成如下命令，修改控制文件中的datafile为实际的路径和名称
    alter system set standby_file_management=manual;
    alter database rename file '/path/bank/kiss/data/oradata/kiss/system.313.762819463' to '/path/bank/kiss/data/oradata/kiss/system01.dbf';
    …省略部分…
    alter database rename file '/path/bank/kiss/data/oradata/kiss/btrsidx.311.762818937' to '/path/bank/kiss/data/oradata/kiss/btrsidx02.dbf';
    alter system set standby_file_management=auto;
5. 如上完成后，重启MRP进程，观察已经正常传输并apply日志。

分析总结

实际上，这个问题主要是因为生产用的ASM，而DG用的文件系统。 由于DG于两天前做了回滚点并open了数据库，后面同事在恢复DG同步时可能重新在生产生成了standby controlfile，但是没有修改datafile的路径，导致MRP进程找不到文件而报错，解决思路就是将控制文件的datafile路径修改正确，使MRP进程能找到文件即可。 这个案例中，还有一种处理方法就是用RMAN，大家可以研究下，此处不介绍了。
