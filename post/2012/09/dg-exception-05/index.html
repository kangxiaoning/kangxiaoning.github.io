解决方法

这里要解决如下几个问题：

  1. 上海DG和同城DG通信异常
  2. 生产向同城DG归档异常
  3. 如上两个问题处理好后，保证从生产到上海的整个级连DG同步正常
  4. 通过RMAN适当清理同城DG和上海DG的FRA，本次我清理了4天前的日志，确认是DG不需要的

分析总结

处理这个问题比较简单，用到的知识在之前发过的DG异常案例都有，因此没有把详细命令和步骤贴出来。

在同城DG和上海DG都正常同步后，我发现在生产切换日志时，不会立刻被同城DG感知，所以现在这个状态是个不合格的同城DG。仔细研究后，发现通过下面操作可以解决该问题：

1
2
3
4
5

1. select * from v$archive_dest where status != 'INACTIVE';  #  这个返回结果发现dest_3依然被oracle认为是invalid，并且错误信息和之前一样
2. alter system set log_archive_dest_state_3 = 'enable';
3. select * from v$archive_dest where status != 'INACTIVE';  #  再次查询，结果显示正常
4. 在生产库切换日志，同城DG立刻有感知，至此这个问题处理完毕

解决问题后，再深入检查了一下，发现同城DG最近一直是通过 fetch gap 的机制和生产同步，因此同步也是延迟的。
