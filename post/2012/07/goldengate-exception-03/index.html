etries exceeded.ING     R_TCIMS     00:00:00      00:00:24    

2012-01-26 07:40:56  ERROR   OGG-01668  PROCESS ABENDING.**************

解决方法

  1. 在源数据库查看该进程参数文件，找到目标数据库
  2. 登录目标数据库，检查GoldenGate进程，发现MGR进程状态为down
  3. 在目标数据库启动MGR进程
  4. 在目标数据库启动其它子进程
  5. 在源数据库启动报错进程
  6. 观察一段时间，进程正常，该问题解决

分析总结

本案例中，根据报错信息“TCP/IP error 239 (Connection refused)”，可以判断在源连接到目标时出现异常，源是主动发起连接的，因此怀疑目标数据库存在异常，进而登录目标数据库检查，很快发现问题并解决。

实际上，这个问题是因为有主机重启，DBA同事在主机重启后没有启动GoldenGate，而目标数据库由于是以测试环境对待，没有配置自动重启的crontab，因此出现上面的异常。

问题现象 OGG-00730

报错内容

1
2
3

2012-01-20 02:55:32  ERROR   OGG-00730  No minimum supplemental logging is enabled. This may cause extract process to handle key update incorrectly if key column is not in first row piece.

2012-01-20 02:55:32  ERROR   OGG-01668  PROCESS ABENDING.

解决方法

  1. select supplemental_log_data_min from v$database; –NO
  2. alter database add supplemental log data;
  3. select supplemental_log_data_min from v$database; –YES
  4. 启动GoldenGate进程，观察正常

分析总结

这个错误发生的背景是生产库重建了控制文件，因此判断是重建控制文件后，没有打开minimum supplemental log，而这是GoldenGate正常同步需要的条件，因此重新打开即可。

问题现象 GGS 218

报错内容

1

2011-09-22 12:25:35  GGS ERROR       218  Error mapping from RSDATA.CIF2_TEMP_CUSTXN to ELISDATA.ECIF_CIF2_TEMP_CUSTXN.

解决方法

  1. info trandata ELISDATA.ECIF_CIF2_TEMP_CUSTXN
  2. 上一步确认附加日志是disabled状态
  3. Add trandata ELISDATA.ECIF_CIF2_TEMP_CUSTXN
  4. 启动进程，观察正常

分析总结

同步update和delete需要Supplemental log，它可以保证源端的redo logfile写入了主键信息供GoldenGate捕获update和delete操作并同步，如果没有打开Supplemental log，replicat进程在同步update和delete时由于没有主键值，无法识别唯一的行，因此将报错。 如下：

  1. 从报错信息获取目标库表上的主键：IDRSDATA.PK_IDRS_HS_BANK_REMAIN_AMOUNT
  2. 在目标执行下面SQL，获取组成主键的列

2
3
4

select *
  from dba_ind_columns
 where index_owner = ' IDRSDATA'
   and index_name = ' PK_IDRS_HS_BANK_REMAIN_AMOUNT ';
  1. 从discard文件获取主键列的值
  2. 主键列为条件，查询目标库的表上是否有记录
  3. 上一步查询表明记录已经存在，所以GoldenGate报主键冲突
  4. 联系运营确认数据是否有问题

分析总结

这个错误很明显，从报错信息可以直接判断出表里的数据存在主键冲突，因此我们直接将冲突的记录拿出来，联系运营，让运营同事配合处理数据即可。

Process: r_lpms_mdm
Time: 2012-01-16 01:18:06
Event Text: Oracle GoldenGate Delivery for Oracle, r_lpms_mdm.prm:  PROCESS ABENDING.

解决方法

  1. 源端重新生产def文件
  2. 拷贝新的def文件到目标端的dirdef目录中
  3. 目标端启动R进程
  4. 观察同步正常

分析总结

这个原因多数是在源端进行了DDL操作，导致表定义改变，但是并没有在目标端重新生成def文件，所以GoldeGate报错。可以通过对源和目标表的LAST_DDL_TIME来判断确实是DDL操作导致。解决方法也很简单，将新的def文件copy到目标端即可。
