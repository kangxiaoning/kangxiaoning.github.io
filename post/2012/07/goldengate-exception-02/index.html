案例整理

目的

对工作中遇到的问题进行总结，记录处理问题的思路和关键步骤，后续遇到类似问题便于参考及迅速处理。

版本下发

这一部分主要是总结版本下发过程中遇到的问题。

问题现象 Trailfile中无logscn

版本下发过程中为了初始化数据，需要在trailfile中获取scn号，按文档步骤正确操作获取scn时，没有得到scn号。

报错内容

无报错，trailfile中没有logscn部分内容。

解决方法

  1. 恢复GoldenGate为正常同步状态
  2. 在ggmgr.t_test_conn_for_gg表里插入一条数据
  3. 按手册中的步骤从头再来一次，重新获取scn
  4. 正常获取scn

分析总结

GoldenGate重启后会跳到下一个trail文件，如果没有数据同步，则不会记录scn号，因此只需要同步一条数据即可。

问题现象 ORA-1466

版本下发过程中按scn号初始化数据时报错。

报错内容

ORA-01466: unable to read data - table definition has changed

解决方法

  1. 检查表的DDL时间，确认是否在初始化过程中发生
  2. 如果DDL时间是在初始始化过程中变化的则联系开发DBA确认是否有关联DB版本
  3. 如果有关联DB版本，需要恢复同步状态，等DB版本部署完成后再下发GoldenGate版本
  4. 如果没有关联DB版本，则恢复同步状态，重新按手册下发GoldenGate版本

分析总结

对表上的DDL操作，例如drop/modify column, move table, drop partition, truncate table/partition等操作，会导致该表执行DDL时间点之前的undo data失效，因此无法获取DDL时间点之前的scn对应的数据，这也是“ORA-01466”错误的原因。

理解了这个原理，也就能明白前面的解决方法了。

对于9i的版本，由于5分钟问题，DDL时间点开始5分钟内的scn也会报上述错误，只需要等待5分钟后再获取新的scn即可解决问题。

问题现象 ORA-08181

版本下发过程中按scn初始化数据时报错。

报错内容

1
2
3
4
5
6
7

SQL> CREATE TABLE DMLBAK.DEPT_INFO_GGBAK    AS SELECT * FROM CHNLDATA.DEPT_INFO     AS OF SCN 8666951057807 WHERE BUSINESS='LBS';

CREATE TABLE DMLBAK.DEPT_INFO_GGBAK    AS SELECT * FROM CHNLDATA.DEPT_INFO     AS OF SCN 8666951057807 WHERE BUSINESS='LBS'

ORA-08181: specified number is not a valid system change number

SQL>

解决方法

  1. 恢复GoldenGate为正常同步状态
  2. 在ggmgr.t_test_conn_for_gg表插入一条数据
  3. 按手册中的步骤从头再来一次，重新获取scn
  4. 再次执行as of scn语句正常

分析总结

由于flashback query是有限制的，最多只能闪回查询5天内的数据，而scn和时间其实是有对应关系的，因此scn有一个最大边界和5天的限制对应，所以当指定的scn离当前时间已经超过5天，则会报上述错误，表示这个scn超过了边界，是无效的。

知道了这个原理，我们需要做的就是让GoldenGate把有效范围内的scn号记录到trailfile即可，这样我们获得的scn就符合flashback query的条件了，因此我们通过在ggmgr.t_test_conn_for_gg表插入一条数据便解决了该问题。
