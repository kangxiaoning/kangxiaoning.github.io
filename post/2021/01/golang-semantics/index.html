Golang value semantics and pointer semantics

在Golang里，一个变量可以是T这种value类型，也可以是*T这种pointer类型，变量的使用从语义上来分，T类型对应value semantics，*T类型对应pointer semantics，对于reference type来说本身就有pointer指向underlying type，因此T类型也有pointer semantics。

概述

Golang既有value semantics又有pointer semantics，特别是作为Method的receiver时如何选择，对开发人员来说经常容易混淆。总的来说有两方面的考虑因素，成本和功能，成本对于性能有影响，功能对设计、可阅读性、可维护性有影响。

成本方面

  • value semantics利用stack管理内存，主要是利用rsp寄存器记录stack可重用内存地址，简单高效
  • pointer semantics需要用到heap内存，Golang对heap内存是通过GC管理的，涉及到比较复杂的逻辑，成本相对较高

功能方面

  • value semantics有隔离修改的功能，因为value semantics是copy要使用的数据，在隔离的内存空间(stack frame)修改只会影响当前函数
  • pointer semantics有共享数据的功能，通过传递pointer来实现，因为共享，所以修改对所有引用均可见，但对于内存占用较大的数据来说，能效提升内存使用率

下面通过对stack的理解，Golang类型，语言设计方面来了解semantics的使用原则及影响。

Stack

“Value semantics keep values on the stack, which reduces pressure on the Garbage Collector (GC). However, value semantics require various copies of any given value to be stored, tracked and maintained. Pointer semantics place values on the heap, which can put pressure on the GC. However, pointer semantics are efficient because only one value needs to be stored, tracked and maintained.” - Bill Kennedy

Types

从本质上来说，编程解决的是数据问题，只有理解具体的数据，我们才能理解要解决的问题，才能在各种权衡中做出合理的设计。

“If you don’t understand the data, you don’t understand the problem. This is because all problems are unique and specific to the data you are working with. When the data is changing, your problems are changing. When your problems are changing, the algorithms (data transformations) needs to change with it.” - Bill Kennedy

以数据驱动的方式思考问题，理解数据对于选择semantics非常有帮助，而Type作为数据在编程中的表现形式，其作用尤为重要，编译器需要根据类型确保数据的完整性。对编译器来说Type承载两个信息，一是数据大小，在内存中需要分配多大的空间；二是如何解释，在计算机内部所有信息都是0/1组成的字节，这些字节对程序员来说，是数字还是字符串或者其它类型，就依靠类型来判断了。

选择semantics在定义类型的时候就应该考虑清楚，因此我们从类型角度看下如何选择semantics，通过了解Golang自身在semantics的选择有肋于理解设计者的选择意图。

我们看下在Golang里以 number, textual, boolean为代表的build-in类型是如何选择semantics的。

Reference Types

Golang里的slice, map, interface, function**, channel都属于reference type**，它们也应该使用value semantics**，在语言设计上也是让这些类型存储在stack里以减轻GC回收heap`内存的压力。

如下是Golang编译器对reference type使用value semantics的一些示例。

func (ip IP) Mask(mask IPMask) IP {
    if len(mask) == IPv6len && len(ip) == IPv4len && allFF(mask[:12]) {
        mask = mask[12:]
    }
    if len(mask) == IPv4len && len(ip) == IPv6len && bytesEqual(ip[:12], v4InV6Prefix) {
        ip = ip[12:]
    }
    n := len(ip)
    if n != len(mask) {
        return nil
    }
    out := make(IP, n)
    for i := 0; i < n; i++ {
        out[i] = ip[i] & mask[i]
    }
    return out
}

IP和IPMask都是基于slice的，也就是它们都是reference type，也遵循了使用value semantics的原则。需要注意的是Mask虽然是个修改数据的操作，但是也使用了value semantics，这里对于有些人认为的“修改数据就用pointer semantics”的错误观点是个较好的证明。

Pointer semantics

func (ip *IP) UnmarshalText(text []byte) error {
	if len(text) == 0 {
		*ip = nil
		return nil
	}
	s := string(text)
	x := ParseIP(s)
	if x == nil {
		return &ParseError{Type: "IP address", Text: s}
	}
	*ip = x
	return nil
}

User Defined Types

这里主要是指定义struct时对semantics的选择，从golang编译器自身在这方面的选择，可以看到也是以value semantics为主，pointer semantics作为例外。如果使用value semantics影响到了正确性，或者不合理，比如数据修改需要被应用到单个值并且被共享，那应该使用pointer semantics。

  	nsec := t.nsec + int32(d%1e9)
  	if nsec >= 1e9 {

使用原则

一致的semantics对于代码的可读性、可维护性非常重要，试想在多人合作开发的情况下，如果大家对semantics的选择不一致，会是多么混乱的局面。在code review过程中关注semantics一致性是必要的，通过统一semantics使大家保持统一的思维模型(mental model)对于维护代码是有利的，随着code base越来越大，这些影响会越来越明显，semantics一致性也越来越重要。

如下是参考文章的作者总结的指导原则，值得学习。

  • At the time you declare a type you must decide what semantic is being used.
  • Functions and methods must respect the semantic choice for the given type.
  • Avoid having method receivers that use different semantics than those corresponding to a given type.
  • Avoid having functions that accept/return data using different semantics than those corresponding to the given type.
  • Avoid changing the semantic for a given type.
  • There are a few exceptions to these guidelines with the biggest being unmarshaling. Unmarshaling always requires the use of pointer semantics. Marshaling and unmarshaling seem to always be exceptions to the rule.
