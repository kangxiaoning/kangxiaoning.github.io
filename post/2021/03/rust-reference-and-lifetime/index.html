Rust reference and lifetime

Rust里的reference与pointer有什么区别？lifetime的作用是什么？为什么只看到reference需要写lifetime参数？针对这些疑问，本篇文章期望能做些总结加深理解。

Scope

了解编译原理里语法分析及语义分析涉及的Scope概念，对于理解Rust里的lifetime有很大的帮助。简单来说Scope就是作用域，表示变量可以有效使用的范围，从程序员的视角看就是一段代码，在这段代码里变量可以正常使用，出了这段代码变量将不能使用，所以这段代码就是该变量的Scope。Rust使用的是 Lexically scoped (or statically scoped) ，变量的作用域和语法相关，看代码一般可以判断出来Scope。如果从编译器角度来理解会更清晰一些，编译器在对程序做语法分析和语义分析过程中，会建立一个树结构来存储变量的作用域信息，这个本质上也代表了变量的lifetime。其实在编译器的角度来看， 每个变量都有lifetime ，也就是生命周期，Rust不使用GC就可以管理内存本质上也是利用了这个特点。

变量在Scope里是语言的预期的行为，必须始终满足这个预期程序才能正常工作，lifetime本质上解决的也是这个问题。

Reference

Rust里的reference与pointer有什么区别？

简单来说reference会被编译器检查，来确保指针始终是有效的，避免出现C/C++里的dangling pointer之类的问题，如果编译器检查通过就可以确保在运行期该指针始终是有效的，如果编译器检查不通过说明存在指针无效的情况。

前面一篇文章介绍ownership时，提到String、HashMap之类的数据结构自身是包含指针的，它们拥有对这些指针的ownership，而reference不一样，reference并不会因为指向了某个对象就对其拥有ownership，所以reference不影响referent的生命周期，反而是reference必须在referent的生命周期内使用，否则就会出现reference无效的情况，不是预期行为。

Rust的reference有两种类型

  1. shared reference ，表示只读引用，在该reference生命周期里可以同时存在多个shared reference

  2. mutable reference ，排它引用或者可修改引用，在该reference生命周期里只能存在一个mutable reference，避免多方写导致数据不一致的情况发生

Lifetime

前面提到每个变量都是有生命周期的，这点很重要，不写lifetime参数并不代表它没有生命周期，而是编译器可以自动完成lifetime参数的补全。

那为什么只看到涉及到reference的地方需要写lifetime参数？

这个原因也很简单，如果变量自身有对其代表的资源的ownership，Rust依靠 Lexically scoped 信息就可以判断变量的生命周期，确保在Scope里的变量始终是符合预期的，在超出Scope后释放资源结束其生命，所以除reference之外的变量不需要额外的lifetime参数就可以完成这个目标。而reference因为多了一层引用，且 reference不拥有referent的ownership ，只靠Scope信息无法确保在reference的Scope里referent始终有效，因此需要lifetime参数来辅助判断 在reference的Scope里referent是否有效 ，以这个判断结果来衡量reference的使用是否符合预期，是否应该让编译器通过检查。比如虽然存储reference的变量是在Scope内，可以正常使用，但是其指向的对象已经不在该Scope或者说已经被销毁，这时候这个reference必然是无效的，不能按预期安全使用，为了解决reference这个隐患，需要程序员做一些辅助，给reference标识下lifetime参数，帮助编译器完成reference有效的判断。

简单来说，除了reference/pointer类型，其它类型只靠Rust的 Lexically scoped 信息就可以判断生命周期，可以确保在生命周期里变量一定是可用的，因此不需要额外的lifetime参数。

Lifetime约束

reference始终有效需要如下两个约束，以下面代码为例。

  fn main() {
      let r;
      {
          let x = 1;
          r = &x;
      }
      assert_eq!(*r, 1);
  }

约束一

  • 从reference自身来看，reference的生命周期不能超出其指向的对象，即 &x 不能超出 x 的生命周期。这个很好理解，如果指向的对象已经销毁了，那这个reference肯定无效了，为了避免这种情况所以需要有这个约束。

约束二

  • 从reference使用来看，reference的生命周期必须大于等于其使用范围，即 &x 的生命周期必须大于等于 r 的生命周期。这个也很好理解，如果reference的生命周期小于reference的使用范围，那在reference使用过程中肯定会出现失效问题，这个约束是为了确保reference在使用过程中始终有效。

第一个约束限制了reference的最大生命周期，第二个约束限制了reference要满足的最小生命周期。

reference在某些情况也不显式写lifetime参数也可以正常工作，这是因为编译器可以根据规则推断出reference要有效使用所需要的lifetime。而有些场景编译器无法完成推断就需要借助程序员来辅助完成，比如函数f入参有多个reference(假设变量名是a, b, c)，返回值也是reference(假设变量名是r)这种场景，如果要确保r始终有效，那我们必须满足上面提到的约束，那么问题来了.

约束分析

  • 如何满足第一个约束？

    • 分析：r肯定不能指向f的本地变量，因为函数运行结束后本地变量就销毁了，r再指向它肯定是无效的。那怎么判断r是不是满足第一个约束？答案是依靠入参信息，前面提到每个变量都是有生命周期的，那么可以让r参照某个入参的生命周期，程序员只要指定r参照某个入参的生命周期就可以帮助Rust编译器来判断是否满足第一个约束。

  • 如何满足第二个约束？

    • 分析：r的使用范围，Rust是可以获取到的，拿r的使用范围对应的生命周期和第一个约束里r自身的生命周期进行比较，就可以判断r是否满足约束，最终这个比较转化成了r的生命周期和r绑定的入参的生命周期之间的比较，因为r和入参的生命周期Rust是可以获取到的，因此最终可以完成r是否有效的判断。
