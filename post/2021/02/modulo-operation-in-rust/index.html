Rust 中允许使用 as 关键字显式转换整数。比如 65535_u32 as i16 将32位的无符号整数转换位16位的有符号整数，目标类型比源类型位数少，必然出现 overflow 了，丢失精度了，那最终的结果是什么？结果是确定的吗？是由什么决定的？

要知道这个结果，我们需要了解背后的原理，也就是取模操作的算法，因为目标位长小于源位长时 Rust 是根据modulo操作得到最终的转换结果。

wikipedia

参考 wikipedia2 关于取模操作的解释，取模操作与除法相关，常见的除法有三种算法，如下图所示分别是Truncated Division，Floored Division，Euclidean Division。不同编程语言采用的算法不一样，比如Rust/C/C++/Java/Go/JavaScript/bash/PHP等采用Truncated，Python/Ruby等采用Floored算法，可以看到大部分编程语言采用的是Truncated算法，也有个别熟知的语言采用Floored算法，这篇文章我们主要介绍这两种算法。

Floored Division

  • 除数为 正 数时

    • 商是负数，此时被除数必然是负数，余数是 正 数

    • 商是正数，此时被除数必然是正数，余数是 正 数

  • 除数为 负 数时

    • 商是负数，此时被除数必然是正数，余数是 负 数，

    • 商是正数，此时被除数必然是负数，余数是 负 数

  • 结论

    • 余数的符号和 除数 一致

    • 商向下取整，即向负无穷方向取整，也就是找到比被除数稍小的最大整数，再计算差值

fn main() {

    // Conversions that are out of range for the destination
    // produce values that are equivalent to the original modulo 2^N,
    // where N is the width of the destination in bits. This
    // is sometimes called "truncation".
    assert_eq!(1000_i16 as u8, 232_u8);
    assert_eq!(65535_u32 as i16, -1_i16);

    assert_eq!(-1_i8 as u8, 255_u8);
    assert_eq!(255_u8 as i8, -1_i8);
}

如上注释所示，在 Rust 的整数类型转换过程中，如果目标类型比源类型的位长小（比如u8的位长是8，i16的位长是16，8<16），转换产生的结果等于 源 mod 2^N 。因此 1000_i16 as u8 的结果是 1000 % 2^8 。根据前面对取模算法的解析，我们可以将上述整数类型转换代入Truncated Division算法的求余公式得到结果。

如下的 int(m / n) 表示 Python 里的 truncate 函数语法。

1000_i16 as u8
  = 1000 % 2^8
  = 1000 % 256
  = 1000 - 256 * int(1000 / 256)
  = 1000 - 256 * 3
  = 232

65535_u32 as i16
  = 65535 % 2^16
  = 65535 % 65536
  = 65535 - 65535 * int(65535 / 65536)
  = 65535 - 0
  = 65535

-1_i8 as u8 因为源和目标位数一样，因此不涉及取模转换，只需要按二进制解释即可，因为 -1_i8 的二进制为 11111111 ，而 11111111 表示为 u8 即为255。

因为在源和目标位长不一致时， Rust 的整数类型转换遵循了Truncated Division规则，所以行为是可预测的，结果是确定的，不像有些语言结果不可预测，这也是 Rust 类型安全的一个体现，即不会出现未定义的行为或者结果。
